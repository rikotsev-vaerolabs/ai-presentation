Slide 2 | Neural Networks

Forward Pass, Loss, Backward Pass

A neural network is a stack of differentiable transformations with trainable parameters. Training adjusts those parameters so the outputs become less wrong.

Forward Pass

Inputs become vectors or matrices. Layers repeatedly apply matrix multiplication, bias addition, normalization, nonlinearities, and attention.

h1 = xW1 + b1; a1 = GELU(h1)
logits = a1W2 + b2
softmax(logits) = probabilities
cross_entropy(probabilities, target) = loss

Logits are raw scores before the final softmax. Inside the MLP, the activation is usually GELU, SiLU, or a gated variant such as SwiGLU.

Backward Pass

  • For every trainable parameter, compute its gradient: dL/dParameter.
  • The gradient says how changing that parameter would change the loss.
  • Backpropagation applies the chain rule from the loss backward through each layer.
  • Then the optimizer nudges parameters opposite the gradient: parameter = parameter - learning_rate × gradient.

Common Layer Types

Embedding: token id → vector Linear: matrix projection Activation: GELU / SiLU nonlinearity Normalization: LayerNorm / RMSNorm Attention: mix allowed context positions Softmax: scores → distribution Residual: preserve signal through depth Dropout: optional training regularizer

Slide 3 | Tokenization

Tokens, Batches, Context

For teaching, use character tokens. Production LLMs usually use subword tokens, but the tensor idea is the same.

1. Split the Data Before Training

The split protects you from fooling yourself. The model should not repeatedly see the final evaluation set while you are still making architecture, hyperparameter, or training decisions.

Training

Used for gradient updates. The model directly learns from these examples.

Validation

Used during development to compare checkpoints, tune hyperparameters, and detect overfitting.

Test

Used only when you believe the model is ready. Repeated test-set checks leak information and increase overfitting risk.

2. Tokens and Batches

A token is the atomic unit the model reads and predicts. Here we use character-level tokenization for the word modern. _ is a special start / padding token and <END> marks the end of the word. With context length 8, each row predicts the next character.

Batch rowTokenized input contextTarget token
0________m
1_______mo
2______mod
3_____mode
4____moder
5___modern
6__modern<END>

Many rows together form a 2D batch matrix: batch_size × context_length. Context length is the maximum number of input tokens the model can directly see in one forward pass.

3. Embeddings Add the Third Dimension

A token embedding matrix has shape vocabulary_size × embedding_dim. Looking up each token id turns the 2D batch into a 3D tensor: batch_size × context_length × embedding_dim. Position information is added separately with positional embeddings or encodings, so the model can distinguish the first m from a later m.

Each token position is represented by a vector of many scalars. This table shows only four channels; real LLMs often use thousands.

Token IDsB × T
Integer ids from the tokenizer.
EmbeddingsB × T × C
Each token-position pair becomes C scalars.
Self-AttentionT ← allowed T
Each position reads from other allowed positions in the same sequence.

In multi-head self-attention, the model splits or projects the representation dimension C across heads. It is not only positional information that is split; token meaning and position are mixed in the representation each head reads.

4. Reading the Shape: B × T × C

The model processes a tensor with three dimensions during training and inference. The context available to one example is mainly its T token positions, represented with C channels at each position. B is parallelism: multiple independent contexts processed together.

B = Batch

How many independent examples are processed at once. Batch items do not see each other as context.

T = Tokens

How many token positions are in each example. This is the context length the model can attend over.

C = Channels

How many scalar features represent each token position after embedding and layer transformations.

From Model to Agent Harness

In product and coding tools, the deployable agent is usually the model plus a harness. The harness supplies memory, tools, permissions, routing, and execution loops.

User

Agent Harness

LLM Core

No persistent memory or external capabilities by itself. It predicts the next token from the tokens currently in context.

Conversation Memory

  • Current message history
  • In-memory buffer
  • Database-backed sessions
  • Summaries loaded into context

Long-Term Memory

  • Retrieved by tools, then inserted into context
  • SQL / document DBs
  • Vector DBs and RAG
  • User profiles or project notes

Tools

  • Web search / scraping
  • Bash execution
  • File editing
  • APIs, browsers, calendars, databases

Slide 5 | Prompt Hierarchy

Prompts Guide; Harnesses Enforce

Prompts are text in the model context. Real constraints come from surrounding software: parsers, sandboxes, approval gates, credentials, and tool schemas.

System / Developer Prompt

Highest-priority behavior, safety, style, and tool-use instructions.

User Prompt

The task request and preferences for the current interaction.

Assistant Tool Request

The model proposes a structured action, such as running a command or calling an API.

Tool Response

Untrusted observed output returned to context for the next reasoning step.

Skills and Sub-Agents in Coding Tools

  • Many coding agents keep skill or agent descriptions as markdown files.
  • The harness or model detects a relevant capability, loads the selected instructions into context, and may expose extra tools.
  • Agent-like behavior is also taught during supervised fine-tuning with examples of planning, tool calls, observations, and final answers.
  • Prompt hierarchy guides behavior, but cannot physically enforce access control. The harness must enforce file permissions, network access, command approval, and tool schemas.

Slide 6 | Model Training Path

How Base Models Become Assistants

The public assistant experience is usually several stages after raw next-token pretraining.

1

Base Model

  • Trained with next-token prediction on massive text/code data.
  • Learns language, style, patterns, and factual associations.
  • Not yet reliably helpful, safe, or instruction-following.
2

SFT

  • Supervised fine-tuning on curated demonstrations.
  • Often human-authored or human-annotated examples.
  • Teaches assistant format, tool-call traces, refusals, and task completion.
3

Preference Signal

  • Human or AI preferences compare candidate answers.
  • A reward model may learn which outputs are better.
  • This is commonly preference ranking, not just a single binary token.
  • Direct methods like DPO can optimize preferences without a separate reward model stage.
4

Preference Optimization

  • Optimize the assistant against preference or reward signals.
  • RLHF uses human preference feedback.
  • RLAIF uses AI-generated feedback or critiques.
  • Modern pipelines may also use DPO, distillation, rejection sampling, or verifier feedback.

Result

The final system is a trained model plus an inference stack, safety policies, tool interfaces, memory systems, and product-specific harness logic. The model may encode factual associations, but retrieval is imperfect and must be checked for stale or invented claims.
